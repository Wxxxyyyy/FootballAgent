# 双向通信：启动顺序与连通性检查

## 一、启动顺序（按这个顺序来）

### 1️⃣ 先启动：本地电脑（中继站）

在**本地电脑**上：

```bash
# 进入项目目录（或你放 relay_api.py 的目录）
cd /path/to/your/relay

# 激活环境（如有）
# conda activate football

# 启动中继站，监听 5000 端口
python relay_api.py
```

看到类似输出表示成功：
```
🔄 启动中继站...
   服务器地址: http://localhost:52335
   OpenClaw地址: http://192.168.10.3:6000
   监听端口: 5000
```

**注意**：`relay_api.py` 里的 `OPENCLAW_URL` 要改成你**旧电脑**的 WiFi IP；`SERVER_URL` 保持 `http://localhost:52335`（Cursor 隧道）或按你实际隧道端口改。

---

### 2️⃣ 建立反向隧道（服务器 → 本地电脑）

要让**服务器**能访问**本地电脑**的 5000 端口，需要在**本地电脑**上执行一次反向 SSH（只需建一次，断开会话会断，可考虑用 `screen`/`tmux` 保活）：

```bash
# 在【本地电脑】执行，把 server_user@server_ip 换成你服务器账号和 IP/域名
ssh -R 15000:localhost:5000 server_user@server_ip
```

含义：服务器上的 `localhost:15000` 会转发到你这台电脑的 `localhost:5000`（中继站）。

- 若你用的是 Cursor 的「反向端口转发」且已经将「远程 15000 → 本地 5000」配好，可跳过这步。
- 若端口 15000 被占用，可改成别的，例如 `25000`，并同步改**服务器**上 `server_api.py` 里的 `RELAY_URL = "http://localhost:25000"`。

---

### 3️⃣ 再启动：服务器

在**服务器**上：

```bash
cd /path/to/footballAgent
conda activate football
python api/server_api.py
```

看到类似输出表示成功：
```
⚽ 服务器 API 启动 | 中继站: http://localhost:15000
```

---

### 4️⃣ 最后启动：旧电脑（OpenClaw 端）

在**旧电脑**上：

```bash
# 进入你放 openclaw_client.py 的目录
cd /path/to/your/openclaw

# 启动服务，监听 6000 端口
python openclaw_client.py
```

看到类似输出表示成功：
```
🔧 启动 OpenClaw 数据端...
   中继站地址: http://192.168.10.5:5000
   监听端口:   6000
```

**注意**：`openclaw_client.py` 里的 `RELAY_URL` 要改成你**本地电脑**的 WiFi IP（和上面中继站那台机器一致）。

---

## 二、连通性检查

### 方法 A：在服务器上检查全链路（推荐）

服务器启动后，在**另一台机器**用浏览器或 `curl` 访问（把 `服务器IP或域名` 换成实际地址）：

```bash
# 检查服务器 + 中继 + OpenClaw 是否都通
curl http://服务器IP或域名:8000/test/connection_status
```

返回示例（都通时）：
```json
{
  "server": "running",
  "relay": "connected",
  "openclaw": "connected",
  "relay_url": "http://localhost:15000",
  "timestamp": "..."
}
```

若 `relay` 为 `disconnected`：检查反向 SSH 是否在本地电脑执行、端口 15000 是否一致。  
若 `openclaw` 为 `disconnected`：检查旧电脑是否已启动 `openclaw_client.py`，以及本地中继里 `OPENCLAW_URL` 的 IP 和端口是否正确。

---

### 方法 B：在服务器上只测“服务器 → OpenClaw”反向通道

```bash
curl http://localhost:8000/test/ping_openclaw
```

应返回 OpenClaw 的 `pong` 信息，说明反向链路（服务器→中继→旧电脑）正常。

---

### 方法 C：在旧电脑上测“OpenClaw → 服务器”正向通道

在**旧电脑**上另开一个终端：

```bash
python openclaw_client.py test
```

会往中继发几条测试数据（实时赔率、批量数据、伤病报告），中继会转发到服务器。  
看**本地电脑**中继终端是否出现 `➡️ 正向转发 -> 服务器`，**服务器**终端是否出现 `⚡ 实时赔率`、`📁 批量数据` 等日志；若都有，说明正向通道正常。

---

## 三、小结

| 顺序 | 在哪里运行       | 命令 / 动作 |
|------|------------------|-------------|
| 1    | 本地电脑         | `python relay_api.py` |
| 2    | 本地电脑         | `ssh -R 15000:localhost:5000 user@server`（或用 Cursor 反向转发） |
| 3    | 服务器           | `python api/server_api.py` |
| 4    | 旧电脑           | `python openclaw_client.py` |
| 检查 | 任意能访问服务器的机器 | `curl http://服务器:8000/test/connection_status` |
| 检查 | 旧电脑           | `python openclaw_client.py test`（测正向） |

把上述顺序跑一遍，再按「二」做一次连通性检查，就能确认双向是否都通。
